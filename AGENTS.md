# AGENTS.md

## Правила Рабочего Процесса

Перед началом любой новой задачи:

1. Забери актуальное состояние репозитория с remote.
2. Убедись, что локальный `main` совпадает с последним `origin/main`.
3. Убедись, что работа не начинается из ветки, которая уже ушла вперед относительно `main`.
4. Если пользователь явно не попросил иное, создай новую feature-ветку от свежего `main`.

## Коммиты

1. Каждый важный логический блок работы должен коммититься отдельно.
2. Используй только conventional commits.
3. Не складывай в один коммит много несвязанных изменений.

Примеры:

- `feat(viewer): add image metadata panel`
- `fix(pdf): preserve page rotation during merge`
- `docs(roadmap): update iteration 2 scope`
- `test(converter): cover heic to jpg pipeline`

## Комментарии В Коде

1. Сложные или ключевые места реализации нужно комментировать на русском.
2. Комментарии должны объяснять, зачем существует логика или какое неочевидное ограничение она учитывает.
3. Не добавляй шумные комментарии к тривиальному коду.

## Визуальный Стиль И UI Foundation

Текущий визуальный стиль проекта уже задан в `frontend/src/App.vue` и `frontend/src/styles.css`. Будущие агенты должны развивать его последовательно, а не заменять на новый по своему вкусу.

1. Базовое направление интерфейса: `soft industrial neumorphism` / современный `neumorphic big-tech dashboard`.
2. Визуальная база должна оставаться светлой, тёплой и тактильной: sand/off-white поверхности, мягкая глубина через свет и тень, без ухода в плоский default SaaS UI.
3. Цвета нужно брать из существующего бренд-направления Jack:
   - глубокий `teal` / `green-black` как основной тёмный тон
   - тёплый `amber` / `coral` как акцент
   - нейтральная светлая sand-база для поверхностей и фона
4. При добавлении новых экранов, секций и компонентов нужно в первую очередь переиспользовать существующие CSS-переменные, радиусы, тени и паттерны поверхности из `frontend/src/styles.css`, а не придумывать новые без причины.
5. Не ломай ключевой характер foundation:
   - крупные выразительные блоки вместо мелкой дробной сетки
   - большие навигационные плитки и сильная визуальная иерархия
   - кастомные SVG/векторные иллюстрации или графические ассоциации для продуктовых модулей
   - аккуратные мягкие hover/focus/active состояния с читаемым контрастом
6. Нельзя скатываться в generic шаблонность:
   - не заменяй текущий стиль на обычные белые карточки с синими кнопками
   - не вводи случайные акцентные цвета, которые не связаны с логотипом
   - не упрощай интерфейс до безличного “admin panel” вида, если задача не требует этого явно
7. Типографика должна сохранять текущую идею: выразительный display для крупных заголовков и спокойный, плотный sans-serif для основного интерфейсного текста.
8. Любые новые UI-решения обязаны быть адаптивными: desktop, tablet и mobile должны проектироваться сразу, а не дорабатываться постфактум.
9. Если в задаче нужно расширить дизайн-систему, сначала расширяй foundation-токены и общие паттерны, затем уже строй конкретный экран.
10. Если пользователь явно не просит радикальный редизайн, изменения нужно встраивать в уже существующее направление, сохраняя преемственность.

## Чеклист Перед Завершением

Перед завершением задачи:

1. Прогони все релевантные тесты.
2. Прогони линтеры и форматирование для затронутых частей.
3. Обнови документацию и roadmap, если изменился scope или поведение.
4. Сделай коммит.
5. Запушь ветку в remote.
6. Подготовь изменение к MR, если пользователь явно не попросил обратное.

## Безопасность

1. Не переписывай историю, если пользователь явно не попросил об этом.
2. Не делай force push без явного разрешения.
3. Не откатывай несвязанные пользовательские изменения.
